Systematic Debugging

Enforces the Iron Law: no fix without confirmed root cause. Five phases — investigation, scope lock, pattern analysis, hypothesis testing (3-strike rule), and implementation with a mandatory failing test first.

How it works

Plain Markdown. Zero dependencies.

01

Install the skills

Each skill is a SKILL.md file. Claude Code reads them on launch and registers the slash commands automatically.

02

Type a command

Claude loads the protocol for that skill and begins executing — reading files, running bash, calling APIs — exactly as defined.

03

Get structured output

Every skill produces a defined report format. No vague prose — findings have severity, location, and remediation steps.

04

State stays on disk

Research briefs, debug reports, and audit findings are written to your filesystem. No external service, no account, no lock-in.

Design principles

Built on four rules

Protocol over personality

Each skill is a deterministic protocol. The audit runs the same six phases every time. The debugger enforces the Iron Law every time. No drift.

No security theatre

The audit looks at git history, CI configs, and dependency trees — not just your application code. Confidence gates suppress noise.

No sycophancy

Reviews and debug reports state findings plainly. No praise, no filler, no "great question". Evidence first, then conclusion.

Evidence before fixes

The Iron Law: no fix without a confirmed root cause. Reviews grep for actual usage before recommending additions. No guessing.
