  <behavior>
    - actionToPrompt maps GsdAction to correct GSD slash command string:
      - plan(1) -> "/gsd:plan-phase 1"
      - execute(1) -> "/gsd:execute-phase 1"
      - verify(1) -> "/gsd:verify-work"
      - resume -> "/gsd:resume-work"
    - runLoop iterates: determineNextAction -> actionToPrompt -> runGsdCommand -> check thresholdHit -> loop
    - When thresholdHit=true, runLoop runs a pause-work session ("/gsd:pause-work") then continues loop (fresh session picks up from disk state)
    - When determineNextAction returns done, runLoop exits normally
    - When determineNextAction returns error, runLoop logs error and exits with code 1
    - SIGTERM handler sets isShuttingDown flag, aborts controller, and after current session completes runs pause-work then exits 0
    - After each runGsdCommand, loop re-reads disk state (calls determineNextAction again) -- no GSD state in daemon memory
  </behavior>
  <action>
    1. Create src/index.ts:

       a. `actionToPrompt(action: GsdAction): string` -- maps action types to GSD slash commands:
          - plan -> `/gsd:plan-phase ${action.phase}`
          - execute -> `/gsd:execute-phase ${action.phase}`
          - verify -> `/gsd:verify-work`
          - resume -> `/gsd:resume-work`
          - done/error -> throw (should not be called with these)

       b. `async function runCheckpoint(config: RunnerConfig): Promise<void>` -- runs a short session with prompt "/gsd:pause-work" using a fresh AbortController. Logs checkpoint start/end.

       c. `async function runLoop(config: RunnerConfig, deps?: { determineNextAction, runGsdCommand, runCheckpoint }): Promise<void>` -- the main loop:
          - Accept optional deps parameter for dependency injection in tests (defaults to real implementations)
          - Create AbortController for current session
          - Loop:
            1. Call determineNextAction(config.projectDir)
            2. If done -> log and return
            3. If error -> log error and throw
            4. Convert action to prompt via actionToPrompt
            5. Call runGsdCommand(prompt, config, controller)
            6. If result.thresholdHit -> call runCheckpoint(config) (this saves context; next iteration will re-read disk state and get a fresh session)
            7. If isShuttingDown -> call runCheckpoint(config) and return
            8. Create new AbortController for next iteration
            9. Continue loop

       d. SIGTERM handler (Pattern 4 from research):
          ```typescript
          let isShuttingDown = false;
          let currentController: AbortController;

          process.on('SIGTERM', () => {
            if (isShuttingDown) return;
            isShuttingDown = true;
            logger.info('SIGTERM received, completing current operation...');
            currentController.abort();
          });
          ```

       e. `async function main(): Promise<void>`:
          - Call loadConfig()
          - Log startup with config summary (projectDir, maxTurns, compactionThreshold)
          - Call runLoop(config)
          - Log completion
          - process.exit(0)
