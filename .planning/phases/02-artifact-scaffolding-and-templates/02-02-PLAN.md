**templates/cortex/clarify-brief.md** (ART-01 template)
Fields: SLUG, TIMESTAMP, STATUS (draft|approved), IDEA (verbatim input), GOAL (outcome statement), NON_GOALS (explicit exclusions), CONSTRAINTS (hard limits), ASSUMPTIONS (things assumed true without verification), OPEN_QUESTIONS (list — each must be actionable), NEXT_RESEARCH_STEPS (ordered list of what to research)

**templates/cortex/research-dossier.md** (ART-02 template)
Fields: SLUG, PHASE (concept|implementation|evals), TIMESTAMP, DEPTH (quick|standard|deep), SUMMARY (one paragraph), FINDINGS (key findings — list), TRADE_OFFS (options considered and tradeoffs), RECOMMENDATIONS (what to do based on findings), OPEN_QUESTIONS (list — carry forward to next phase if unresolved), SOURCES (list of sources consulted)

**templates/cortex/spec.md** (ART-03 template)
All 9 required sections must appear (omitting any is an error):
1. PROBLEM — what is being built and why (one paragraph)
2. SCOPE — in-scope list AND out-of-scope list (separate subsections)
3. ARCHITECTURE_DECISION — the chosen approach and why (with alternatives considered)
4. INTERFACES — external interfaces this spec touches (APIs, contracts, module boundaries)
5. DEPENDENCIES — libraries, services, other cortex artifacts this depends on
6. RISKS — list of risks with mitigation per risk
7. SEQUENCING — order of implementation steps (numbered list)
8. TASKS — discrete implementation tasks (each task gets a checkbox)
9. ACCEPTANCE_CRITERIA — measurable, testable criteria (each gets a checkbox)

Status field: draft | approved | superseded

**templates/cortex/gsd-handoff.md** (ART-04 template)
This is a GSD-ready work order. The human imports this into GSD explicitly — Cortex never calls GSD commands.
Required sections: OBJECTIVE (what must be built and why), DELIVERABLES (list of artifacts to produce), REQUIREMENTS (requirement IDs from the project REQUIREMENTS.md that this work satisfies), TASKS (ordered implementation tasks with checkboxes), ACCEPTANCE_CRITERIA (measurable done criteria matching the active contract), CONTRACT_LINK (relative path to docs/cortex/contracts/SLUG/contract-001.md)

Write the 3 remaining artifact templates. The contract template is the most critical: it must include `eval_plan` (mandatory per docs/EVALS.md — contracts without it are invalid).

**templates/cortex/contract.md** (ART-05 template)
Header block:
```
# Contract: {SLUG} — {PHASE}

**ID:** {CONTRACT_ID}
**Slug:** {SLUG}
**Phase:** {PHASE}
**Created:** {TIMESTAMP}
**Status:** draft | approved | closed
```
Required sections (all mandatory):
- OBJECTIVE — single clear statement of what this contract delivers
- DELIVERABLES — list of artifacts to be produced
- SCOPE — in-scope list AND out-of-scope list (separate subsections)
- WRITE_ROOTS — paths that the executing agent is allowed to write to; any write outside these roots is a violation
- DONE_CRITERIA — measurable, testable criteria; each gets a checkbox
- VALIDATORS — list of validation commands/checks; each gets a checkbox
- EVAL_PLAN — path to eval plan: `docs/cortex/evals/{SLUG}/eval-plan.md`
  - Add comment: `<!-- Required. Contract is incomplete without this field. -->`
- APPROVALS — two checkboxes: Contract approval and Evals approval, both initially unchecked
- ROLLBACK_HINTS — steps to reverse this contract's changes if needed

**templates/cortex/eval-proposal.md** (ART-06 template)
Fields: SLUG, TIMESTAMP, STATUS (draft|approved|rejected)
Required sections:
- PROPOSED_DIMENSIONS — list from the 8-candidate matrix: functional correctness, regression, integration, safety/security, performance, resilience, style, UX/taste; each item notes whether `approval_required: true`
- FIXTURES — test fixtures or data needed for each proposed dimension
- RUBRICS — scoring criteria per dimension
- THRESHOLDS — pass/fail thresholds per dimension (e.g., "all assertions pass", ">95% coverage", "no P0 findings")
- FAILURE_TAXONOMY — categories of failures and their severity (P0/P1/P2/P3)
- APPROVAL_REQUIRED flag at document level (boolean) — `true` if any dimension requires human approval (UX/taste always requires it; high-stakes and ambiguous evals require it)
- REVIEWER — who must approve (human name or role)
- APPROVAL_STATUS — pending | approved | rejected

**templates/cortex/eval-plan.md** (ART-07 template)
Written after human approval of eval-proposal. Fields: SLUG, TIMESTAMP, APPROVED_BY, APPROVED_AT
Required sections:
- APPROVED_DIMENSIONS — subset of proposed dimensions that were approved (list)
- FIXTURES_PER_DIMENSION — fixtures for each approved dimension (subsection per dimension)
- THRESHOLDS_PER_DIMENSION — pass/fail threshold per dimension
- RUN_INSTRUCTIONS — how to execute this eval plan (ordered steps)
- RESULTS — populated after running (initially empty: checkboxes, one per dimension)
