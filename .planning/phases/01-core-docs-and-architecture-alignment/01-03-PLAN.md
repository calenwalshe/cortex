**FILE 1: docs/EVALS.md**

This document describes evals as a first-class lifecycle component. It is a specification of the intended eval subsystem (Phase 5 deliverable). Write in present-tense imperative as an architectural spec, not a tutorial.

Required sections:

1. **Why Evals Are First-Class** — Two short paragraphs. Every active contract must reference an eval plan. Evals are not an afterthought — they are the gate between assure and done. Silent failures are not allowed: a failed eval must produce a repair recommendation or open a repair contract.

2. **Eval Lifecycle** — Sequential flow with one paragraph per step:
   - Proposal: `/cortex-research --phase evals` produces an eval proposal covering dimensions, fixtures, rubrics, thresholds, and failure taxonomy
   - Human approval: when an eval is subjective, high-stakes, or ambiguous, an explicit human approval is required before the plan is written — system blocks progression until answered
   - Plan: after approval, eval-plan.md is written
   - Execution: evals run against the contract's deliverables
   - Results: pass → advance to assure; fail → repair recommendation or repair contract
   - Repair/Assure: repair loop re-runs evals; all evals passed is the gate to done

3. **Candidate Eval Matrix** — Table with all 8 dimensions from REQUIREMENTS.md EVAL-05 (use these exact names):
   | Dimension | What It Covers | When Mandatory |
   - Functional correctness: does the implementation do what the spec says?
   - Regression: did this break existing behavior?
   - Integration: do the components work together correctly?
   - Safety/security: does the implementation introduce security vulnerabilities?
   - Performance: does it meet the performance requirements in the contract?
   - Resilience: does it handle failure modes correctly?
   - Style: does it follow the project's coding and documentation conventions?
   - UX/taste: does the output feel right to a human? (subjective — always requires human approval)

4. **When Human Approval Is Mandatory** — Bullet list:
   - Any eval dimension marked as subjective in the eval proposal
   - Any eval where the success threshold is ambiguous
   - Any eval for a high-stakes or irreversible operation (data migrations, auth changes, security boundaries)
   - UX/taste dimension always requires human approval
   - When the eval-designer agent marks approval_required: true in the eval proposal

5. **Repair Loop** — Short section: failed eval → repair recommendation written to docs/cortex/ → if repair is complex, a new repair contract is opened → repair implemented → evals re-run. Silent failure is never acceptable.

6. **Invocation** — How to invoke:
   - Eval proposal: `/cortex-research --phase evals`
   - Eval plan: written after human approves the proposal
   - Evals run: triggered by cortex-validator-trigger hook (Phase 4) or manually

7. **Artifact Paths** — Table:
   | Artifact | Path (in target repo) |
   - Eval proposal: `docs/cortex/evals/<slug>/eval-proposal.md`
   - Eval plan: `docs/cortex/evals/<slug>/eval-plan.md`
   - Eval results: `docs/cortex/evals/<slug>/results-<timestamp>.md` (produced at execution time)

   Add: "All eval artifacts are written to the target project repo, not the Cortex framework repo."

8. **Contract Reference Requirement** — Short section: every active contract (docs/cortex/contracts/<slug>/contract-001.md) must include an eval_plan field pointing to the eval-plan.md path. Contracts without an eval plan reference are incomplete.

**FILE 2: docs/AGENTS.md**

This document is the Phase 4 specification for the 4 Cortex agents. Write as an architectural spec, not current state. Add a note at the top: "The agents described here are Phase 4 deliverables. They are not yet installed. This document defines the intended agent specifications."

Required sections:

1. **Agent Roster** — Overview table:
   | Agent | Role | Write Scope | Read Scope | Mode |
   - cortex-specifier: drafts specs and contracts from research; write to docs/cortex (specs/, contracts/); read all; sub-agent
   - cortex-critic: adversarial reviewer of specs, contracts, decisions; write none; read all; sub-agent
   - cortex-scribe: maintains continuity artifacts; write to docs/cortex and .cortex; read all; sub-agent
   - cortex-eval-designer: proposes eval suites, rubrics, fixtures, thresholds; write to docs/cortex/evals/; read all; sub-agent

2. **Per-Agent Specifications** — One H3 section per agent with:
   - Role description (2-3 sentences)
   - Tools available (read/write file tools, bash access, etc.)
   - Write permission scope (exact paths)
   - Read permission scope
   - Invocation: how it is called (sub-agent mode, or via --team flag in /cortex-research)
   - Output artifacts: what it produces

   cortex-specifier: Drafts specs and contracts from research dossiers. Does not do the research — reads existing dossiers and clarify briefs. Write scope: docs/cortex/specs/, docs/cortex/contracts/. Produces: spec.md, gsd-handoff.md, contract-001.md.

   cortex-critic: Read-only adversarial reviewer. Reviews specs, contracts, and architectural decisions for logical gaps, missing edge cases, and incorrect assumptions. Produces: a critique report (written to docs/cortex/reviews/ or returned inline). Never writes directly to specs or contracts.

   cortex-scribe: Maintains all continuity artifacts. Runs at session transitions, compaction events, and phase transitions. Write scope: docs/cortex/handoffs/, .cortex/. Produces: current-state.md updates, next-prompt.md refreshes, decisions.md log entries.

   cortex-eval-designer: Proposes eval suites. Reads the spec and contract, proposes rubrics, fixtures, and thresholds for all relevant dimensions. Write scope: docs/cortex/evals/. Produces: eval-proposal.md.

3. **Permission Model** — Short explanation:
   - Write-restricted agents (specifier, scribe, eval-designer) can only write to their designated paths
   - Read-only agents (critic) cannot write any files
   - No agent writes to .planning/ — GSD owns all planning state
   - No agent writes outside docs/cortex/ or .cortex/ unless explicitly listed above

4. **Invocation** — How agents are called:
   - Sub-agent mode: invoked by Cortex commands when needed (e.g., /cortex-spec invokes cortex-specifier)
   - Team mode: `/cortex-research --team` invokes the full agent team for research
   - Direct invocation via `@cortex-specifier` etc. is supported

5. **Installation** — Path: `~/.claude/agents/`. Installer (Phase 6) symlinks agent YAML files from the Cortex repo into this path.
