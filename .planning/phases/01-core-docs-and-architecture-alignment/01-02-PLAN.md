Create docs/COMMANDS.md as a new file. This is the operator reference for all 7 /cortex-* commands.

Structure: One H2 section per command, in spine order (clarify, research, spec, investigate, review, audit, status). Each section must have these subsections:

**Per-command template:**
- **Syntax** — exact invocation syntax with arguments and flags in bash code block
- **Purpose** — one sentence: what this command does and why you'd use it
- **Inputs** — table or list of: argument/flag, required/optional, description, default
- **Outputs** — table or list of: artifact name, path (exact docs/cortex/ path from REQUIREMENTS.md), what it contains
- **Rules** — bullet list of constraints, gate conditions, and behavioral rules (e.g., phase-guard restrictions, approval gates)
- **Example** — a realistic example invocation

Use vNext flag conventions throughout — these come from REQUIREMENTS.md, NOT from the current SKILL.md files (which predate vNext). The current cortex-research SKILL.md uses --quick/--deep and writes to ~/research/; COMMANDS.md must document the vNext interface.

Content per command:

**1. /cortex-clarify**
- Syntax: `/cortex-clarify <idea>`
- Purpose: Converts a fuzzy idea into a written problem frame (clarify brief)
- Inputs: `<idea>` — required positional — the idea, problem, or feature to clarify (quoted string or inline text)
- Outputs: clarify brief at `docs/cortex/clarify/<slug>/<timestamp>-clarify-brief.md` (in target repo). Sections: goal, non-goals, constraints, assumptions, open questions, next research steps.
- Rules: Creates the slug from the idea text. Does not start research or spec. Clarify brief is the gate to /cortex-research.
- Example: `/cortex-clarify "add smart retry logic to the API client"`

**2. /cortex-research**
- Syntax: `/cortex-research [<topic>] [--phase concept|implementation|evals] [--depth quick|standard|deep] [--team]`
- Purpose: Produces a research dossier for the current slug at a specified phase and depth
- Inputs:
  - `<topic>` — optional — focus topic (defaults to current slug's clarify brief)
  - `--phase` — optional — concept (default), implementation, or evals
  - `--depth` — optional — quick, standard (default), or deep
  - `--team` — optional flag — invokes agent team for research (opt-in, adds cost)
- Outputs: dossier at `docs/cortex/research/<slug>/<phase>-<timestamp>.md` (in target repo)
- Rules: Reads clarify brief as input context. Each --phase produces a separate dossier. --phase evals produces an eval proposal (see /cortex-audit and docs/EVALS.md). Human must explicitly request each phase.
- Example: `/cortex-research --phase implementation --depth deep`

**3. /cortex-spec**
- Syntax: `/cortex-spec`
- Purpose: Compresses clarify brief + research dossier(s) into a spec, GSD handoff, and first execution contract
- Inputs: None (reads docs/cortex/clarify/ and docs/cortex/research/ for current slug)
- Outputs:
  - spec.md at `docs/cortex/specs/<slug>/spec.md` — schema: problem, scope, architecture decision, interfaces, dependencies, risks, sequencing, tasks, acceptance criteria
  - gsd-handoff.md at `docs/cortex/specs/<slug>/gsd-handoff.md` — GSD-ready work-order
  - contract-001.md at `docs/cortex/contracts/<slug>/contract-001.md` — schema: id, slug, phase, objective, deliverables, scope, write roots, done criteria, validators, approvals, rollback hints
- Rules: Requires clarify brief to exist. Requires at least one research dossier. Does NOT auto-invoke GSD — the human explicitly imports gsd-handoff.md into GSD. Spec and contract must be human-approved before execution begins.
- Example: `/cortex-spec`

**4. /cortex-investigate**
- Syntax: `/cortex-investigate [<subject>]`
- Purpose: Writes an investigation artifact and can hand off into a GSD repair contract
- Inputs: `<subject>` — optional — what to investigate (defaults to current active contract context)
- Outputs: investigation artifact(s) at `docs/cortex/investigations/<slug>/` (in target repo)
- Rules: Typically invoked after a validator failure or unexpected behavior. Can produce a repair contract handed off to GSD.
- Example: `/cortex-investigate "rate limiter not triggering in test environment"`

**5. /cortex-review**
- Syntax: `/cortex-review [<target>]`
- Purpose: Writes a review artifact including a contract compliance lens
- Inputs: `<target>` — optional — file, PR, or component to review (defaults to current active contract scope)
- Outputs: review artifact(s) at `docs/cortex/reviews/<slug>/` (in target repo). Includes contract compliance section.
- Rules: Review always checks the active contract's done criteria and validators. Output is an artifact — not a chat-only response.
- Example: `/cortex-review src/api/retry.ts`

**6. /cortex-audit**
- Syntax: `/cortex-audit [<target>]`
- Purpose: Writes a security and quality audit artifact covering required lenses
- Inputs: `<target>` — optional — scope to audit (defaults to current active contract write roots)
- Outputs: audit artifact(s) at `docs/cortex/audits/<slug>/` (in target repo)
- Rules: Must cover all 7 required lenses: authentication, data handling, secrets exposure, unsafe tool usage, input validation, dependency risks, misuse vectors. No lens may be omitted without explicit note.
- Example: `/cortex-audit src/`

**7. /cortex-status**
- Syntax: `/cortex-status`
- Purpose: Reconstructs current working context from repo-local artifacts and updates continuity handoff files
- Inputs: None (reads .cortex/state.json, docs/cortex/ artifacts, current-state.md)
- Outputs: Updated continuity files (current-state.md, next-prompt.md). Terminal summary of current slug, mode, open questions, blockers, next action.
- Rules: Safe to run at any time. Designed for use after /clear or compaction. Does not require chat history. Always reads from repo-local artifacts, never from chat. Primary recovery command.
- Example: `/cortex-status`

Add a brief intro section at the top of the file: "All commands write artifacts to the target project repo (the repo where Cortex is installed and used). The Cortex framework repo itself is not modified by command invocations."

Add a "Flag Reference" section at the bottom with a quick-reference table: Flag | Commands | Values | Description.

Create docs/CONTINUITY.md as a new file. This document answers: "What is Cortex's continuity strategy, what artifacts does it maintain, and how do I resume after /clear?"

Required sections:

1. **Why Continuity Matters** — Short section: Claude's chat history is ephemeral. /clear, /compact, and context exhaustion destroy it. Cortex solves this by writing all intelligence state to repo-local artifacts that persist across sessions. /cortex-status is the recovery command.

2. **The Continuity Stack** — Describe the three-layer stack:
   - Repo artifacts (always-on): docs/cortex/ and .cortex/ files written by commands
   - Session hooks (Phase 4): cortex-session-start hydrates on SessionStart, cortex-precompact/postcompact surround /compact, cortex-session-end writes on exit
   - Machine state: .cortex/state.json tracks runtime mode, artifacts, approvals, gates
   Note clearly: "Session hooks are Phase 4 deliverables — they are not yet active. In the current state, /cortex-status must be run manually after resuming a session."

3. **Continuity Files** — Full list of all 8 files with their locations and purposes:

   **Human-readable (in docs/cortex/handoffs/ of the target project repo):**
   | File | Purpose |
   - current-state.md — current working state snapshot
   - open-questions.md — unresolved questions that block progress
   - next-prompt.md — paste-ready restart prompt for after /clear
   - decisions.md — architectural and design decisions log
   - eval-status.md — current eval pass/fail state
   - last-compact-summary.md — written by cortex-postcompact hook

   **Machine state (in .cortex/ of the target project repo):**
   | File | Purpose |
   - .cortex/state.json — runtime mode, artifacts, approvals, gates
   - .cortex/compaction/precompact-<timestamp>.md — pre-compaction snapshots

4. **current-state.md Schema** — Field-by-field definition:
   | Field | Type | Description |
   - slug: string — current active slug (e.g., "retry-logic-v2")
   - mode: enum — clarify | research | spec | execute | validate | repair | assure | done
   - approval_status: enum — pending | approved | rejected
   - active_contract_path: string — relative path to contract-001.md for current slug
   - recent_artifacts: array — list of artifact paths written in current session
   - open_questions: array — questions that must be resolved to advance
   - blockers: array — hard blockers preventing phase transition
   - next_action: string — the single recommended next step

5. **next-prompt.md Format** — Explain: next-prompt.md contains a short paragraph (3-5 sentences) that a human can paste after /clear to restore context. It names the current slug, mode, what was last accomplished, what the next step is, and where the active contract is. Example structure (not a real example — show the template shape).

6. **.cortex/state.json Schema** — JSON example with comments:
   ```json
   {
     "slug": "current-active-slug",
     "mode": "spec",
     "approval_status": "pending",
     "active_contract": "docs/cortex/contracts/<slug>/contract-001.md",
     "artifacts": ["list of written artifact paths"],
     "approvals": {"contract": false, "evals": false},
     "gates": {"spec_complete": true, "contract_approved": false}
   }
   ```

7. **Resume Protocol** — Numbered steps:
   1. Open the target project repo
   2. Run `/cortex-status`
   3. Cortex reads current-state.md, next-prompt.md, and the active contract
   4. Terminal output shows: current slug, mode, open questions, blockers, next action
   5. Paste next-prompt.md contents if more context is needed
   6. Continue from where you left off

8. **Compaction Flow** — Describe (this is Phase 4 behavior, mark as such):
   - PreCompact hook: writes snapshot to .cortex/compaction/precompact-<timestamp>.md, refreshes current-state.md
   - /compact runs
   - PostCompact hook: writes compact summary to last-compact-summary.md, refreshes next-prompt.md

9. **Repair and Assure Continuity** — Short note: after each repair loop iteration, continuity artifacts are updated (LOOP-03). eval-status.md tracks which evals passed/failed. /cortex-status can be run mid-repair to get current state.
